# 分支模型与安全钩子（贡献指南）

本项目采用 **develop 为集成主干** 的分支流程，并通过 Git pre-push 钩子拦截敏感信息泄露。**克隆仓库后请先执行「一次性启用」。**

---

## 一、一次性启用（clone 后必做）

Git 默认不会自动应用仓库内的钩子，需手动把 hooks 路径指向 `.githooks/`：

```bash
git config core.hooksPath .githooks
chmod +x .githooks/*      # macOS / Linux
```

执行后，本仓库的所有钩子即对你本地生效。

---

## 二、分支模型

| 分支 | 角色 | 部署环境 | 谁能直接改 |
| --- | --- | --- | --- |
| `main` | 生产主干 | **生产**（`topic.yeranyang.com`） | 仅通过 develop 合入 |
| `develop` | **集成/预览主干** | **预览**（EdgeOne 自动分配预览 URL） | 仅通过 feature/bugfix 合入 |
| `feature/*` | 新功能 | 预览（推送即建预览） | 开发者 |
| `bugfix/*` | 缺陷修复 | 预览 | 开发者 |

### 流程

```
feature/xxx ──┐
              ├──(合并)──▶ develop ──(验收后合并)──▶ main
bugfix/xxx  ──┘             预览环境                  生产环境
```

1. 从 `develop` 切出工作分支：`git switch -c feature/xxx develop`
2. 开发、提交，推送（触发预览构建自检钩子）
3. 合并回 `develop`：验证预览环境
4. 预览验收通过后，由 `develop` 合并进 `main` 发布生产

> **EdgeOne 多环境**：`main` 独占生产，其余任何分支 push 后自动进入预览环境。因此 `develop` 及各 feature 分支推送后都会得到独立预览 URL。KV 为项目级共享，预览验证请**只读不写**。

## 三、pre-push 敏感信息扫描

`.githooks/pre-push` 会在每次 `git push` 前，扫描**本次待推送提交新增的行**，命中以下任一规则即**拦截推送**：

- GitHub PAT（`github_pat_` / `ghp_` / `ghu_` 等）
- OpenAI 密钥（`sk-...`）
- 腾讯云 SecretId（`AKID...`）、AWS Access Key（`AKIA...`）
- Google API Key（`AIza...`）、Slack Token（`xox...`）
- 私钥文件头（`-----BEGIN ... PRIVATE KEY-----`）
- 通用密钥赋值（`secret/password/api_key/access_token = "..."`）

含 `YOUR_`、`example`、`placeholder`、`<...>` 等占位内容的命中视为安全，不拦截。

### 误报时如何绕过

确认确实是误报（例如示例文档里的假密钥），可临时跳过：

```bash
git push --no-verify
```

> ⚠️ 仅在**确认无真实密钥**时使用。真泄露时正确做法是：移除密钥 → 改用环境变量/密钥管理 → 清除历史（`git filter-repo`）→ 到平台**吊销并轮换**该凭证。

### 新增扫描规则

编辑 `.githooks/pre-push` 顶部 `patterns` 数组，按 `"规则名|正则"` 格式追加即可。
